Naming Conventions

Variables & Functions

Use descriptive, meaningful names that clearly indicate purpose.

âœ— Bad const d = new Date(); const x = users.filter(u => u.a); âœ“ Good const currentDate = new Date(); const activeUsers = users.filter(user => user.isActive);

Naming Style Guidelines

  • camelCase - Variables, functions: getUserData(), isActive
  • PascalCase - Classes, components: UserProfile, DataService
  • UPPER_SNAKE_CASE - Constants: MAX_RETRY_COUNT, API_BASE_URL
  • kebab-case - File names: user-profile.js, api-client.ts

Boolean Naming

Prefix boolean variables with is, has, can, or should.

const isAuthenticated = true; const hasPermission = checkPermissions(user); const canEdit = user.role === 'admin'; const shouldRender = isVisible && hasData;

Code Formatting

Indentation & Spacing

  • Use 2 or 4 spaces for indentation (be consistent)
  • Maximum line length: 80-120 characters
  • Add blank lines between logical blocks
  • One statement per line

Brackets & Braces

âœ“ Good - K&R Style if (condition) { doSomething(); } else { doSomethingElse(); } function calculateTotal(items) { return items.reduce((sum, item) => sum + item.price, 0); }

Imports & Dependencies

Group and order imports logically:

// 1. Standard library imports import { useState, useEffect } from 'react'; // 2. Third-party imports import axios from 'axios'; import _ from 'lodash'; // 3. Internal/local imports import { UserService } from './services'; import { formatDate } from './utils';

Code Structure

Function Length

  • Keep functions small and focused (ideally under 50 lines)
  • Each function should do one thing well
  • Extract complex logic into separate functions

DRY Principle (Don't Repeat Yourself)

âœ— Bad - Repetition function calculateTaxForUS(amount) { return amount * 0.07; } function calculateTaxForCA(amount) { return amount * 0.05; } âœ“ Good - Reusable function calculateTax(amount, taxRate) { return amount * taxRate; } const TAX_RATES = { US: 0.07, CA: 0.05 };

Early Returns

Reduce nesting with guard clauses and early returns.

âœ“ Good function processUser(user) { if (!user) return null; if (!user.isActive) return null; if (!user.hasPermission) return null; return performProcessing(user); }

Documentation Standards

Code Comments

  • Explain WHY, not WHAT - Code should be self-explanatory
  • Document complex algorithms or business logic
  • Add TODOs with assigned developer and date
  • Remove commented-out code before committing

Function Documentation

/** * Calculates the total price including tax and discount * * @param {number} basePrice - The original price before modifications * @param {number} taxRate - Tax rate as decimal (e.g., 0.08 for 8%) * @param {number} discountPercent - Discount as percentage (0-100) * @returns {number} Final price after tax and discount * @throws {Error} If basePrice is negative */ function calculateFinalPrice(basePrice, taxRate, discountPercent) { if (basePrice < 0) { throw new Error('Price cannot be negative'); } const discountAmount = basePrice * (discountPercent / 100); const priceAfterDiscount = basePrice - discountAmount; return priceAfterDiscount * (1 + taxRate); }

Security Best Practices

Input Validation

  • Never trust user input - Always validate and sanitize
  • Use parameterized queries to prevent SQL injection
  • Validate data types, ranges, and formats
  • Escape output to prevent XSS attacks

Secrets Management

âœ— Bad - Hardcoded secrets const API_KEY = 'sk-1234567890abcdef'; const password = 'myPassword123'; âœ“ Good - Environment variables const API_KEY = process.env.API_KEY; const password = process.env.DB_PASSWORD;

Never commit: API keys, passwords, tokens, certificates

Authentication & Authorization

  • Use secure password hashing (bcrypt, Argon2)
  • Implement proper session management
  • Use HTTPS for all communications
  • Apply principle of least privilege
  • Implement rate limiting

Testing Standards

Test Coverage Requirements

  • Minimum 80% code coverage for critical paths
  • All public APIs must have tests
  • Test edge cases and error conditions
  • Run tests before every commit

Test Structure (AAA Pattern)

test('should calculate discount correctly', () => { // Arrange const originalPrice = 100; const discountPercent = 20; // Act const result = calculateDiscount(originalPrice, discountPercent); // Assert expect(result).toBe(80); });

Testing Best Practices

  • Unit Tests: Test individual functions in isolation
  • Integration Tests: Test component interactions
  • E2E Tests: Test complete user workflows
  • Keep tests fast and independent
  • Use descriptive test names that explain the scenario
  • Mock external dependencies
